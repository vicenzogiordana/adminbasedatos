% Abstract
\begin{abstract}
El auge de la inteligencia artificial generativa y los modelos de lenguaje grandes ha impulsado la adopción de bases de datos vectoriales como infraestructura fundamental para la búsqueda semántica y los sistemas de Retrieval-Augmented Generation (RAG). Qdrant emerge como una solución de código abierto que combina rendimiento en búsquedas aproximadas por vecindad con garantías de durabilidad y flexibilidad en el filtrado de resultados. Este trabajo documenta la arquitectura interna de Qdrant, incluyendo el algoritmo de indexación HNSW, el Write-Ahead Log para recuperación ante fallos, la organización de segmentos con memory mapping, y los mecanismos de concurrencia. A partir de la experiencia práctica con un sistema RAG implementado, se presenta un análisis crítico de las fortalezas, limitaciones y casos de uso de esta tecnología.
\end{abstract}

\section{Introducción}
\label{sec:introduccion}

El panorama de la inteligencia artificial ha experimentado una transformación radical con la proliferación de modelos de lenguaje grandes (LLMs) capaces de generar texto, traducir idiomas, resumir documentos y mantener conversaciones naturales. Sin embargo, estos modelos presentan limitaciones conocidas: tienden a ``alucinar'' información incorrecta, no pueden acceder a datos actualizados en tiempo real, y carecen de mecanismos nativos para referenciar fuentes específicas.

La técnica de Retrieval-Augmented Generation (RAG) surge como respuesta a estas limitaciones, permitiendo que los LLMs consulten bases de conocimiento externas antes de generar respuestas. El componente crítico de estos sistemas es la base de datos vectorial, que almacena representaciones numéricas (embeddings) de documentos, imágenes o cualquier dato no estructurado, habilitando búsquedas por similitud semántica en espacios de alta dimensionalidad.

\subsection{El problema de la búsqueda tradicional}

Las bases de datos relacionales tradicionales fueron diseñadas para almacenar y consultar datos estructurados mediante lenguajes como SQL. Su modelo de indexación (B-trees, hash tables) resulta óptimo para búsquedas exactas por clave primaria o por valores exactos de campos. Sin embargo, presenta limitaciones severas cuando se trata de:

\begin{itemize}
    \item \textbf{Búsqueda por similitud semántica}: no existe forma directa de expresar ``buscar documentos similares a X'' en SQL
    \item \textbf{Datos no estructurados}: texto libre, imágenes, audio no encajan en esquemas rígidos
    \item \textbf{Escalabilidad en dimensionalidad}: los índices tradicionales degradan dramáticamente con vectores de miles de dimensiones
\end{itemize}

\subsection{Embeddings y representación vectorial}

La representación vectorial de datos resuelve estos problemas mediante embeddings: transformaciones de información no estructurada en vectores numéricos en espacios continuos de alta dimensionalidad. Modelos especializados como nomic-embed-text, BERT o CLIP capturan relaciones semánticas en las coordenadas vectoriales:

\begin{equation}
    \text{sim}(\text{embed}("gato"), \text{embed}("felino")) > \text{sim}(\text{embed}("gato"), \text{embed}("coche"}))
    \label{eq:semantic_similarity}
\end{equation}

Esta propiedad permite encontrar contenido relacionado basándose en significado, no en palabras exactas.

\subsection{Qdrant como solución}

Qdrant~\cite{qdrant_docs} es una base de datos vectorial de código abierto escrita en Rust que aborda específicamente las necesidades de aplicaciones modernas de IA. Diseñado desde cero para rendimiento y confiabilidad, ofrece:

\begin{itemize}
    \item Implementación optimizada del algoritmo HNSW para búsquedas aproximadas por vecindad más cercana
    \item Write-Ahead Log para garantizar durabilidad de operaciones
    \item Arquitectura de segmentos con soporte para memory mapping
    \item Filtrado avanzado por payloads combinado con búsquedas vectoriales
    \item API REST y gRPC de alto rendimiento
    \item Despliegue simple mediante Docker
\end{itemize}

\subsection{Objetivo del informe}

El presente trabajo tiene como objetivo documentar la arquitectura interna de Qdrant y evaluar su funcionamiento a través de la implementación de un sistema RAG completo. Se espera que el análisis proporcione una comprensión profunda de cómo las bases de datos vectoriales abordan problemas que las soluciones tradicionales no pueden resolver eficientemente.

\subsection{B. Modelo de datos}

Qdrant organiza los datos en una jerarquía de cuatro niveles:

\begin{enumerate}
    \item \textbf{Tenant (opcional)}: grupo aislable de recursos para multi-tenancy
    \item \textbf{Collection}: contenedor de vectores con configuración de indexación compartida
    \item \textbf{Point}: unidad fundamental consistente en vector, payload e identificador
    \item \textbf{Segment}: archivo de almacenamiento que agrupa points
\end{enumerate}

\subsubsection{Colecciones}

Una collection define el esquema de los vectores almacenados:

\begin{itemize}
    \item \param{size}: dimensionalidad de los vectores (ej: 768 para nomic-embed-text)
    \item \param{distance}: métrica de similitud para comparaciones
    \item \param{vectors\_config}: configuración de vectores (múltiples por punto)
\end{itemize}

\subsubsection{Puntos}

Cada punto contiene:
\begin{itemize}
    \item \code{id}: identificador único (numérico o UUID)
    \item \code{vector}: coordenadas en espacio n-dimensional
    \item \code{payload}: diccionario de pares clave-valor con metadatos
    \item \code{shard\_key}: clave para distribución en modo cluster
\end{itemize}

\begin{lstlisting}[language=json, caption={Estructura de un punto en Qdrant}]
{
    "id": 12345,
    "vector": [0.1, 0.2, 0.3, ...],
    "payload": {
        "text": "contenido del documento",
        "source": "articulo.txt",
        "timestamp": 1718000000,
        "tags": ["tecnologia", "ia"]
    }
}
\end{lstlisting}

\subsubsection{Métricas de distancia}

Qdrant soporta tres métricas de similitud:

\begin{itemize}
    \item \textbf{Cosine}: similitud del ángulo entre vectores
    \begin{equation}
        \text{cosine}(A, B) = \frac{A \cdot B}{\|A\| \cdot \|B\|}
        \label{eq:cosine}
    \end{equation}
    \item \textbf{Dot product}: producto punto normalizado
    \begin{equation}
        \text{dot}(A, B) = \frac{A \cdot B}{\|A\| \cdot \|B\|}
        \label{eq:dot}
    \end{equation}
    \item \textbf{Euclidean}: distancia euclidiana
    \begin{equation}
        \text{euclidean}(A, B) = \sqrt{\sum_{i=1}^{n}(a_i - b_i)^2}
        \label{eq:euclidean}
    \end{equation}
\end{itemize}

La elección de métrica depende del modelo de embedding utilizado y del tipo de datos. Para embeddings normalizados (como nomic-embed-text), la similitud coseno es generalmente la mejor opción.

\subsection{E. Requisitos del sistema}

Qdrant está diseñado para desplegarse principalmente mediante Docker, lo que simplifica significativamente los requisitos de instalación.

\subsubsection{Hardware recomendado}

\begin{itemize}
    \item \textbf{CPU}: 4+ cores (más cores mejora throughput de búsqueda)
    \item \textbf{RAM}: 8+ GB (depende del tamaño de la colección y uso de mmap)
    \item \textbf{Disco}: SSD NVMe recomendado para rendimiento óptimo
    \item \textbf{Red}: 1 Gbps mínimo para deployments distribuidos
\end{itemize}

\subsubsection{Sistemas operativos compatibles}

\begin{itemize}
    \item Linux (Ubuntu 20.04+, Debian 11+, CentOS 8+)
    \item macOS (Apple Silicon y Intel)
    \item Windows (mediante WSL2 o Docker Desktop)
    \item Contenedores: Docker, Podman, Kubernetes
\end{itemize}

\subsubsection{Docker}

El despliegue mínimo requiere Docker 20.10+:

\begin{lstlisting}[language=bash, caption={Despliegue básico con Docker}]
docker run -d --name qdrant \
    -p 6333:6333 \
    -p 6334:6334 \
    -v $(pwd)/qdrant_storage:/qdrant/storage \
    qdrant/qdrant
\end{lstlisting}

El directorio montado se utiliza para persistir datos en disco.

\subsection{F. Ediciones y limitaciones}

La edición open source de Qdrant presenta las siguientes características y limitaciones:

\begin{itemize}
    \item \textbf{Funcionalidades ilimitadas}: todas las funcionalidades core están disponibles sin límites en número de colecciones, puntos o dimensiones
    \item \textbf{Sin límites artificiales}: no hay límites en el tamaño de datos o velocidad de consultas
    \item \textbf{Configuración completa}: acceso a todos los parámetros de HNSW, WAL y optimizers
    \item \textbf{No hay gestión automática}: la escalabilidad, backups y monitoreo son responsabilidad del operador
    \item \textbf{Soporte comunitario}: ayuda disponible a través de GitHub Issues y Discord, sin garantía de tiempo de respuesta
\end{itemize}

\section{Arquitectura y Funcionamiento Interno}
\label{sec:arquitectura}

\subsection{A. Arquitectura del sistema}

La arquitectura de Qdrant se compone de cuatro módulos principales que interactúan para proporcionar un servicio de base de datos vectorial completo y eficiente.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.45\textwidth]{qdrant_architecture.png}
    \caption{Arquitectura de Qdrant: componentes principales y flujo de datos}
    \label{fig:architecture}
\end{figure}

\subsubsection{Capa de API}

Qdrant expone una API dual:

\begin{itemize}
    \item \textbf{REST API}: accesible en el puerto 6333, proporciona endpoints para CRUD de colecciones, puntos y búsquedas
    \item \textbf{gRPC API}: accesible en el puerto 6334, ofrece mejor rendimiento para operaciones intensivas
\end{itemize}

La API REST es preferida para desarrollo y debugging, mientras que gRPC se recomienda para producción con alto throughput.

\begin{lstlisting}[language=bash, caption={Ejemplo de API REST para búsqueda}]
POST /collections/{name}/points/search
{
    "vector": [0.1, 0.2, ...],
    "limit": 10,
    "with_payload": true
}
\end{lstlisting}

\subsubsection{Motor de búsqueda HNSW}

El motor de búsqueda implementa el algoritmo Hierarchical Navigable Small World (HNSW) para búsquedas aproximadas por vecindad más cercana. Este componente es responsable de:

\begin{itemize}
    \item Construcción del índice graph durante ingestas
    \item Búsqueda eficiente de vecinos más cercanos
    \item Balance entre precisión y velocidad mediante parámetros configurables
\end{itemize}

El índice HNSW se construye incrementalmente sobre los segmentos inmutables, permitiendo optimización continua sin bloquear operaciones de lectura.

\subsubsection{Gestor de segmentos}

El gestor de segmentos coordina el almacenamiento físico de datos:

\begin{itemize}
    \item Creación de nuevos segmentos para operaciones de escritura
    \item Transición de segmentos a estado inmutable para optimización
    \item Compactación y fusión de segmentos obsoletos
    \item Aplicación de políticas de memory mapping
\end{itemize}

Los segmentos son archivos independientes que pueden ser accedidos en paralelo por múltiples threads de búsqueda.

\subsubsection{Capa de almacenamiento}

La capa de almacenamiento maneja la persistencia de datos:

\begin{itemize}
    \item Escritura de segmentos a disco
    \item Gestión del Write-Ahead Log
    \item Creación y restauración de snapshots
    \item Coordinación de WAL con segmentos para durabilidad
\end{itemize}

\subsection{B. Modelo de almacenamiento}

Qdrant organiza los datos en disco siguiendo una estructura optimizada para el acceso eficiente y la recuperación ante fallos.

\subsubsection{Estructura de segmentos}

Cada segmento en disco contiene:

\begin{itemize}
    \item \textbf{Header}: metadatos del segmento (versión, número de puntos, etc.)
    \item \textbf{Vector data}: coordenadas de vectores en formato binario optimizado
    \item \textbf{Payload data}: índices y datos de payloads en formato columnar
    \item \textbf{Index data}: estructuras HNSW para búsqueda vectorial
    \item \textbf{Field data}: índices secundarios para filtrado
\end{itemize}

Los segmentos pueden ser de dos tipos:

\begin{enumerate}
    \item \textbf{Appendable}: acepta nuevas operaciones de escritura
    \item \textbf{Index}: inmutable, optimizado para búsquedas con índice HNSW completo
\end{enumerate}

\begin{lstlisting}[caption={Estructura de archivos en disco}]
storage/
├── collections/
│   └── my_collection/
│       ├── snapshots/
│       └── segments/
│           ├── 1234567890_segtemp.dat  # Segmento appendable
│           └── 1234567891_segindex.dat # Segmento inmutable
├── wal/
│   └── 00000.log                       # Write-Ahead Log
└── tmp/                                # Archivos temporales
\end{lstlisting}

\subsubsection{Índice HNSW en disco}

El índice HNSW se almacena de forma independiente al vector data, permitiendo:

\begin{itemize}
    \item Carga perezosa de componentes del índice según necesidad
    \item Reconstrucción parcial del índice sin reindexar todos los vectores
    \item Separación de datos calientes (índice en RAM) y fríos (índice en disco)
\end{itemize}

El archivo de índice HNSW contiene:

\begin{itemize}
    \item Matriz de adyacencia del grafo (punteros a nodos vecinos)
    \item Metadata de capas y niveles
    \item Entry point para búsqueda
\end{itemize}

\begin{equation}
    \text{Memoria índice HNSW} = O(n \cdot m \cdot \log n)
    \label{eq:hnsw_memory}
\end{equation}
donde $n$ es el número de puntos y $m$ es el número de conexiones por nodo.

\subsection{C. Recuperación y durabilidad de datos}

Qdrant implementa múltiples mecanismos para garantizar la integridad y disponibilidad de los datos.

\subsubsection{Write-Ahead Log (WAL)}

El WAL es el componente fundamental para la durabilidad en Qdrant. Su principio de funcionamiento es:

\begin{quotation}
    ``Antes de modificar datos persistentes, registrar la intención de modificar en un log que se sincronice a disco''
\end{quotation}

El WAL de Qdrant registra tres tipos de operaciones:

\begin{itemize}
    \item \textbf{INSERT}: agregar nuevos puntos
    \item \textbf{DELETE}: marcar puntos como eliminados
    \item \textbf{UPDATE}: modificar vectores o payloads
\end{itemize}

\begin{lstlisting}[caption={Estructura de registro WAL}]
struct WALRecord {
    version: u32,           // Versión del formato
    operation: OpType,      // INSERT | DELETE | UPDATE
    sequence_num: u64,      // Número de secuencia
    point_id: u64,          // ID del punto
    vector: Option<Vec<f32>>,// Vector (si aplica)
    payload: Option<HashMap>,// Payload (si aplica)
    checksum: u32,          // Verificación CRC32
}
\end{lstlisting}

La configuración del WAL permite ajustar el compromiso entre durabilidad y rendimiento:

\begin{itemize}
    \item \param{wal\_capacity\_mb}: tamaño máximo antes de rotación (default: 32MB)
    \item \param{wal\_segments\_ahead}: segmentos a escribir por delante
\end{itemize}

La rotación del WAL ocurre cuando se alcanza el límite de tamaño o se completa un checkpoint de consistencia. Una vez que los datos están persistidos en segmentos, el WAL puede truncarse de forma segura.

\begin{enumerate}
    \item Operaciones se escriben en WAL (append-only)
    \item WAL se sincroniza a disco (fsync)
    \item Operaciones se aplican a segmentos
    \item Segmentos se persisten asíncronamente
    \item WAL se trunca hasta el checkpoint
\end{enumerate}

    \centering
    \includegraphics[width=0.4\textwidth]{wal_flow.png}
    \caption{Flujo de datos: WAL y segmentos}
    \label{fig:wal_flow}
\end{figure}

\subsubsection{Snapshots}

Qdrant soporta la creación de snapshots para backup y migración:

\begin{itemize}
    \item Snapshot completo de una colección
    \item Incluye segmentos, WAL y metadatos
    \item Compresión opcional para reducir tamaño
    \item Restauración point-in-time
\end{itemize}

\begin{lstlisting}[language=bash, caption={Creación de snapshot}]
# Crear snapshot
curl -X POST http://localhost:6333/collections/my_collection/snapshots

# Listar snapshots
curl http://localhost:6333/collections/my_collection/snapshots

# Restaurar desde snapshot
curl -X PUT http://localhost:6333/collections/my_collection/snapshots/recover \
    -d '{"location": "s3://bucket/snapshot.snapshot"}'
\end{lstlisting}

\subsubsection{Recuperación ante fallos}

En caso de fallo del sistema, Qdrant ejecuta el siguiente protocolo de recuperación:

\begin{enumerate}
    \item Leer el WAL desde el último checkpoint conocido
    \item Reconstruir el estado de todas las colecciones iterando los registros
    \item Verificar consistencia de segmentos existentes
    \item Aplicar las operaciones pendientes del WAL
    \item Truncar el WAL hasta el punto de sincronización
\end{enumerate}

El tiempo de recuperación es proporcional al tamaño del WAL, no al de la colección completa. Una colección de 10 millones de puntos puede recuperarse en segundos si el WAL es pequeño.

\subsection{D. Gestión de transacciones y concurrencia}

Qdrant implementa un modelo de concurrencia optimizado para workloads de búsqueda vectorial, donde las lecturas superan ampliamente a las escrituras.

\subsubsection{Modelo de threads}

El servidor Qdrant utiliza un pool de threads con la siguiente distribución:

\begin{itemize}
    \item \textbf{API threads}: manejan requests entrantes
    \item \textbf{Write workers}: procesan operaciones de escritura secuencialmente
    \item \textbf{Read workers}: ejecutan búsquedas en paralelo
    \item \textbf{Optimization workers}: compactan y fusionan segmentos
\end{itemize}

La separación entre writers y readers permite que las búsquedas no se bloqueen por escrituras en curso.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.4\textwidth]{concurrency_model.png}
    \caption{Modelo de concurrencia de Qdrant}
    \label{fig:concurrency}
\end{figure}

\subsubsection{Transacciones de escritura}

Las operaciones de escritura siguen este flujo:

\begin{enumerate}
    \item Request llega al thread API
    \item Se valida y se registra en el WAL (durabilidad garantizada)
    \item Se adquiere lock del segmento destino
    \item Se aplica la operación al segmento
    \item Se libera el lock
    \item Se confirma la operación al cliente
\end{enumerate}

El WAL proporciona durability antes de que los datos sean aplicados a segmentos. Esto significa que incluso si el proceso falla antes de aplicar al segmento, los datos se recuperan del WAL al reiniciar.

\subsubsection{Lecturas concurrentes}

Las búsquedas pueden ejecutarse en paralelo sobre múltiples segmentos:

\begin{enumerate}
    \item Query llega a cualquier thread API disponible
    \item Se divide la búsqueda en subtareas por segmento
    \item Read workers ejecutan búsquedas en paralelo
    \item Se fusionan y ordenan los resultados parciales
    \item Se devuelve el resultado consolidado
\end{enumerate}

Los segmentos inmutables pueden ser accedidos concurrentemente sin sincronización, ya que no se modifican durante las búsquedas.

\subsubsection{Modo distribuido (Raft)}

En deployments distribuidos, Qdrant utiliza el algoritmo Raft para consenso:

\begin{itemize}
    \item Un líder acepta todas las escrituras
    \item Réplicas aplican operaciones en el mismo orden
    \item Failover automático si el líder falla
    \item Lecturas pueden servirse desde cualquier nodo (eventual consistency)
\end{itemize}

\section{Ventajas, Limitaciones y Escenarios de Uso}

\subsection{A. Fortalezas técnicas}

Qdrant presenta múltiples fortalezas que lo distinguen en el ecosistema de bases de datos vectoriales:

\begin{enumerate}
    \item \textbf{Rendimiento excepcional}: la implementación en Rust proporciona baja latencia y alto throughput. Las búsquedas HNSW típicamente completan en menos de 10ms para colecciones de millones de puntos.
    
    \item \textbf{Eficiencia de memoria}: el uso de memory mapping permite manejar colecciones que exceden la RAM disponible, con paginación transparente del sistema operativo.
    
    \item \textbf{Filtrado avanzado}: la combinación de búsqueda vectorial con filtrado por payload permite consultas complejas que aprovechan tanto semántica como metadatos estructurados.
    
    \item \textbf{Durabilidad incorporada}: el WAL integrado proporciona garantías de consistencia sin necesidad de configuración adicional.
    
    \item \textbf{Despliegue simple}: la imagen Docker oficial permite poner en marcha un servidor funcional en segundos.
    
    \begin{lstlisting}[language=bash, caption={Despliegue en un comando}]
docker run -p 6333:6333 qdrant/qdrant
    \end{lstlisting}
    
    \item \textbf{API flexible}: soporte para REST y gRPC permite integración con cualquier stack tecnológico.
    
    \item \textbf{Configuración granular}: todos los parámetros de HNSW, WAL y optimizers son ajustables para optimizar según el caso de uso.
\end{enumerate}

\subsection{B. Limitaciones}

A pesar de sus fortalezas, Qdrant presenta ciertas limitaciones que deben considerarse:

\begin{enumerate}
    \item \textbf{No es un motor relacional}: Qdrant no soporta joins, transacciones ACID complejas ni consultas SQL. Es exclusivamente una base de datos vectorial.
    
    \item \textbf{Curva de aprendizaje en parámetros HNSW}: la configuración óptima de $m$, $ef\_construct$ y otros parámetros requiere comprensión del algoritmo y experimentación.
    
    \begin{itemize}
        \item $m$ muy bajo: precisión sacrificada
        \item $m$ muy alto: memoria excesiva
        \item $ef\_construct$ muy alto: construcción lenta
        \item $ef\_construct$ muy bajo: calidad del índice comprometida
    \end{itemize}
    
    \item \textbf{Sin UI de administración nativa}: la gestión se realiza mediante API o herramientas de terceros. No existe un dashboard web incorporado.
    
    \item \textbf{Escalabilidad manual en self-hosted}: la versión open source no incluye auto-scaling. El operador debe gestionar la capacidad.
    
    \item \textbf{Limitado para datos estructurados}: no hay soporte para queries complejas sobre payloads más allá del filtrado básico.
\end{enumerate}

\section{Conclusiones}
\label{sec:conclusiones}

\subsection{Análisis del grupo}

A partir de la experiencia práctica implementando un sistema RAG completo con Qdrant, el grupo ha desarrollado una comprensión profunda de las bases de datos vectoriales y sus aplicaciones en IA generativa.

\subsection{Lo más interesante}

El aspecto más sorprendente de Qdrant fue la implementación del algoritmo HNSW. La combinación de una estructura de grafo jerárquica con búsqueda greedy permite lograr complejidades sub-lineales sin sacrificar excesivamente la precisión. La posibilidad de configurar parámetros como $m$ y $ef\_construct$ para ajustar el trade-off entre velocidad y precisión demuestra la madurez de la solución.

El Write-Ahead Log también resultó más elegante de lo esperado. La combinación de durabilidad con performance (append-only, persistencia asíncrona) es un patrón bien ejecutado que minimiza la latencia de escritura sin comprometer la integridad de los datos.

\subsection{Lo más complejo}

El ajuste de parámetros de HNSW representó el mayor desafío. Encontrar la combinación correcta de $m$, $ef\_construct$, y $ef$ requiere experimentación y comprensión profunda del algoritmo. Un valor incorrecto puede resultar en búsquedas lentas, uso excesivo de memoria, o precisión degradada.

La integración con Ollama también presentó retos, particularmente en la gestión de la latencia de generación de embeddings y respuestas. El procesamiento asíncrono y el manejo de timeouts son aspectos que requieren cuidadosa implementación.

\subsection{Opinión fundamentada sobre Qdrant}

Qdrant demuestra ser una solución robusta y bien diseñada para el almacenamiento y búsqueda de vectores de alta dimensionalidad. Sus principales fortalezas son:

\begin{itemize}
    \item \textbf{Arquitectura limpia}: la separación entre API, motor de búsqueda, gestor de segmentos y almacenamiento facilita el mantenimiento y la extensión.
    
    \item \textbf{Rendimiento consistente}: las pruebas realizadas mostraron latencias predecibles incluso bajo carga concurrente.
    
    \item \textbf{Flexibilidad de filtrado}: la capacidad de combinar búsquedas vectoriales con condiciones sobre payloads habilita casos de uso complejos sin necesidad de capas adicionales.
    
    \item \textbf{Documentación y comunidad}: la documentación oficial es completa y la comunidad en Discord es receptiva y helpful.
\end{itemize}

Como limitaciones, señalamos que Qdrant no debe considerarse como un reemplazo de bases de datos relacionales, sino como un complemento especializado para workloads vectoriales. La gestión de una instalación self-hosted requiere conocimiento de Docker, monitoreo y capacidad de planificación de capacidad.

Para proyectos que requieran búsqueda semántica o sistemas RAG, Qdrant representa una opción excelente que combina rendimiento, confiabilidad y código abierto. La decisión entre la versión self-hosted y Qdrant Cloud dependerá del equipo disponible para operaciones y los requisitos de disponibilidad.

En conclusión, Qdrant cumple su promesa de proporcionar una base de datos vectorial de alto rendimiento accesible para equipos de todos los tamaños. Su adopción creciente y la comunidad activa sugieren que continuará evolucionando para satisfacer las demandas de aplicaciones de IA cada vez más sofisticadas.

// Qdrant client wrapper using native fetch API
const QDRANT_URL = `http://${process.env.QDRANT_HOST}:${process.env.QDRANT_PORT}`;

async function qdrantRequest<T>(path: string, options: RequestInit = {}) {
    const response = await fetch(`${QDRANT_URL}${path}`, {
        ...options,
        headers: { 'Content-Type': 'application/json', ...options.headers }
    });
    if (!response.ok) {
        const error = await response.text();
        throw new Error(`Qdrant error ${response.status}: ${error}`);
    }
    return response.json();
}

// Busqueda de chunks similares
export async function searchChunks(
    queryVector: number[],
    limit: number = 5,
    scoreThreshold: number = 0.4
): Promise<Chunk[]> {
    const result = await qdrantRequest<{
        result: Array<{ id: string; score: number; payload?: Record<string, unknown> }>;
    }>(`/collections/rag_collection/points/search`, {
        method: 'POST',
        body: JSON.stringify({
            vector: queryVector,
            limit,
            score_threshold: scoreThreshold,
            with_payload: true
        })
    });
    return result.result.map(r => ({
        text: r.payload?.text as string || '',
        source: r.payload?.source as string || '',
        chunk_id: r.payload?.chunk_id as number || 0
    }));
}
